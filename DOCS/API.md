# API.md
## Интеграционный контракт Badrudin AI OS

**Версия:** 0.1  
**Статус:** базовая спецификация для проектирования и разработки  
**Проект:** Badrudin AI OS для ООО «Экстра-Элит»  
**Назначение:** определить единые правила обмена данными между CRM, мобильными формами, ИИ-агентами, почтой, мессенджерами, складом, финансами и другими корпоративными системами



## 1. Цель документа

Настоящий документ определяет, как компоненты Badrudin AI OS должны безопасно обмениваться данными и запускать бизнес-процессы.

API должен превратить разрозненные сообщения, письма, документы, фото, видео, голосовые сообщения, заявки, финансовые сведения и отчёты в единый управляемый поток.

Система должна:

- собирать информацию только из разрешённых источников;
- определять отправителя, объект, проект и смысл сообщения;
- распознавать задачи, риски, расходы, заявки и документы;
- предлагать действия ответственным ИИ-агентам;
- требовать подтверждение человека для критических действий;
- передавать утверждённые поручения реальным сотрудникам;
- контролировать сроки, доказательства выполнения и просрочки;
- хранить полную историю действий;
- формировать утренние, оперативные и вечерние сводки.

API не является разрешением на бесконтрольный доступ к личным перепискам или внешним системам. Каждая интеграция включается отдельно, через официальный механизм доступа и с утверждёнными правами.



## 2. Место API в архитектуре

API является единым контрактом между следующими слоями:

1. Веб-интерфейс CRM.
2. Мобильные формы прорабов и сотрудников.
3. Сервис интеграций с почтой и мессенджерами.
4. Оркестратор ИИ-агентов.
5. Модуль задач и контроля исполнения.
6. Проектный и объектовый учёт.
7. Документооборот.
8. Склад, закупки, техника и инструмент.
9. Подотчётные денежные средства и финансовый контроль.
10. Уведомления и эскалации.
11. Аналитика, KPI, аудит и отчётность.
12. Внешние CRM, ERP, бухгалтерские и государственные системы.

Все клиенты используют единые идентификаторы, статусы, правила доступа и журналирование.



## 3. Главные принципы

### 3.1. Единый источник истины

После приёма и проверки данные сохраняются в Badrudin AI OS. Мессенджер, письмо или внешняя CRM остаются источником, но не заменяют внутреннюю запись.

### 3.2. Человек утверждает критические действия

Без подтверждения уполномоченного человека запрещено:

- отправлять юридически значимые документы;
- утверждать платежи и выдачу денежных средств;
- изменять договоры и финансовые обязательства;
- увольнять или нанимать сотрудников;
- менять ключевые сроки проекта;
- принимать технические решения с риском безопасности;
- списывать крупные объёмы материалов;
- публиковать официальные заявления компании;
- передавать конфиденциальные сведения внешним получателям.

### 3.3. Официальные интеграции

Подключения выполняются только через официальные API, вебхуки, корпоративные почтовые протоколы или утверждённые коннекторы.

Запрещается:

- обходить ограничения платформ;
- использовать неофициальный перехват личных переписок;
- хранить пароли сотрудников в открытом виде;
- автоматизировать действия через личные аккаунты без разрешения;
- получать данные из чатов, к которым компания не имеет законного доступа.

Для WhatsApp используются только возможности официальной бизнес-платформы, которые реально поддерживаются и разрешены на момент внедрения. Произвольное чтение личных аккаунтов или неразрешённых групп не допускается.

### 3.4. Минимально необходимые права

Каждый пользователь, агент и коннектор получает только те разрешения, которые нужны для его роли.

### 3.5. Полная проверяемость

Каждый запрос, ответ, автоматическое решение, подтверждение, отправка и изменение статуса должны иметь запись в журнале аудита.



## 4. Архитектурный стиль

Базовый вариант API:

- REST API для операций чтения и изменения данных;
- webhooks для приёма событий из внешних систем;
- очередь событий для фоновой обработки;
- WebSocket или Server-Sent Events для обновлений интерфейса в реальном времени;
- объектное хранилище для фото, видео, голосовых сообщений и документов;
- OpenAPI как машиночитаемое описание REST API;
- версионирование через префикс `/api/v1`;
- JSON как основной формат обмена;
- UTF-8 для всех текстов;
- ISO 8601 для дат и времени;
- UUID для внутренних идентификаторов.

Пример базового адреса:

```text
https://api.badrudin-ai-os.example/api/v1
```

Фактический домен определяется при развёртывании и не должен быть зашит в клиентский код.



## 5. Среды работы

Система должна разделять среды:

- `development` - локальная разработка;
- `test` - автоматические тесты;
- `staging` - проверка перед выпуском;
- `production` - рабочая система.

Данные production запрещено автоматически копировать в development или test без обезличивания и утверждения.

Каждая среда имеет отдельные:

- базы данных;
- ключи шифрования;
- токены интеграций;
- хранилища файлов;
- очереди;
- журналы;
- домены и настройки.



## 6. Идентификация и авторизация

### 6.1. Пользователи

Пользователь входит через корпоративную учётную запись, одноразовый код, пароль с многофакторной защитой или утверждённого поставщика идентификации.

### 6.2. Сервисные учётные записи

ИИ-агенты, фоновые процессы и внешние коннекторы используют отдельные сервисные учётные записи. Запрещено использовать учётную запись директора для фоновой автоматизации.

### 6.3. Токены

Рекомендуется:

- короткоживущий access token;
- refresh token с возможностью отзыва;
- OAuth 2.0 или OpenID Connect для внешних подключений;
- подпись webhook-сообщений;
- ротация секретов;
- хранение секретов в защищённом хранилище.

### 6.4. Обязательные данные контекста

Каждый запрос должен содержать или позволять определить:

- пользователя или сервис;
- организацию;
- роль;
- объект или проект, если применимо;
- источник запроса;
- идентификатор трассировки;
- язык интерфейса;
- часовой пояс.

## 8. Общий формат запроса

Пример заголовков:

```http
Authorization: Bearer <access_token>
Content-Type: application/json
Accept: application/json
X-Request-ID: <uuid>
X-Idempotency-Key: <uuid>
X-Client-Version: <version>
```

`X-Idempotency-Key` обязателен для операций, которые нельзя безопасно выполнить дважды:

- создание платежной заявки;
- выдача подотчётных средств;
- проведение складского движения;
- отправка сообщения;
- создание поручения из внешнего события;
- регистрация webhook-события.



## 9. Общий формат ответа

Успешный ответ:

```json
{
  "success": true,
  "data": {},
  "meta": {
    "request_id": "uuid",
    "timestamp": "2026-07-18T03:00:00+03:00"
  }
}
```

Ответ с ошибкой:

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Проверьте обязательные поля",
    "details": [
      {
        "field": "project_id",
        "reason": "required"
      }
    ]
  },
  "meta": {
    "request_id": "uuid",
    "timestamp": "2026-07-18T03:00:00+03:00"
  }
}
```

В production запрещено возвращать пользователю трассировку стека, секреты, SQL-запросы или внутренние адреса инфраструктуры.



## 10. Коды ошибок

Базовые коды:

- `AUTHENTICATION_REQUIRED`;
- `ACCESS_DENIED`;
- `VALIDATION_ERROR`;
- `RESOURCE_NOT_FOUND`;
- `CONFLICT`;
- `DUPLICATE_EVENT`;
- `RATE_LIMIT_EXCEEDED`;
- `INTEGRATION_UNAVAILABLE`;
- `APPROVAL_REQUIRED`;
- `APPROVAL_REJECTED`;
- `FILE_TOO_LARGE`;
- `UNSUPPORTED_FILE_TYPE`;
- `MALWARE_DETECTED`;
- `BUSINESS_RULE_VIOLATION`;
- `INTERNAL_ERROR`.

Пользовательское сообщение должно быть понятным. Техническая причина сохраняется в журнале с ограниченным доступом.



## 11. Пагинация, фильтры и сортировка

Списки поддерживают:

- `page` и `page_size` либо cursor-based pagination;
- фильтр по статусу;
- фильтр по проекту и объекту;
- период дат;
- поиск по тексту;
- сортировку;

> **Реализовано (общая пагинация списков, §13/§25).** Списочные эндпоинты принимают
> аддитивные параметры `limit` (1..200) и `offset` (>=0). По умолчанию (`limit`
> не задан) возвращается весь список — существующие клиенты не ломаются и данные не
> обрезаются молча; недопустимые значения (`limit`=0/>200, `offset`<0) отклоняются с
> `422`. Срез выполняется после ABAC-фильтрации по доступным объектам. Общий помощник —
> `app.api.pagination` (`page_params`, `paginate`). Подключено к списочным эндпоинтам
> `/integrations`, `/smm`, `/kpi`, `/pto`, `/notifications`, `/risks`, `/inbox`,
> `/agents` (агенты и предложения), `/tasks/control` (просроченные, уведомления),
> `/core` (проекты, задачи, согласования, ежедневные отчёты), `/crm` (контрагенты,
> лиды, сделки, коммуникации), `/warehouse` (остатки, резервы; журнал `/ledger` уже
> ограничен параметром `limit`); остальные списки подключаются поэтапно тем же
> помощником.
- выбор полей;
- включение связанных сущностей только при необходимости.

Пример:

```http
GET /api/v1/tasks?project_id=<uuid>&status=overdue&page=1&page_size=50
```

Ограничения размера страницы устанавливаются сервером.

## 17. Офлайн-синхронизация

Каждая мобильная запись получает локальный идентификатор и время создания.

При восстановлении связи клиент передаёт пакет изменений:

```text
POST /mobile/sync
```

Сервер обязан:

- обнаружить дубликаты;
- сохранить порядок событий;
- не провести финансовую или складскую операцию дважды;
- сообщить о конфликте;
- вернуть карту локальных и серверных идентификаторов;
- не терять файлы при частичной ошибке;
- позволить повторную отправку.

При конфликте финансовых, складских или юридически значимых данных автоматическое объединение запрещено.

## 19. Почтовые интеграции

- Microsoft Graph для утверждённой корпоративной среды;
- Gmail API для утверждённой корпоративной среды;
- IMAP и SMTP только при безопасной конфигурации и отсутствии более подходящего официального API.

Основные операции управления коннектором:

```text

POST   /integrations/email/{connection_id}/test
```

Входящие письма могут автоматически:

Отправка писем от имени сотрудника требует утверждённых правил. Юридически значимые, финансовые и конфликтные ответы должны проходить подтверждение человеком.

- Microsoft Teams;
- Slack;
- другие платформы только при наличии официального API и законного доступа.

## 44. Защита от prompt injection и вредоносных вложений

Текст письма, сообщения или документа считается недоверенным вводом.

ИИ-агент не должен выполнять инструкции, содержащиеся во внешнем сообщении, если они пытаются:

- изменить системные правила;
- получить секреты;
- отправить данные третьей стороне;
- выполнить команду;
- отменить согласование;
- повысить права;
- удалить информацию;
- выдать внешний текст за распоряжение директора.

Перед использованием внешнего содержимого система отделяет данные от команд и применяет политики доступа.



## 45. Ограничение частоты и защита от перегрузки

API применяет лимиты по:

- пользователю;
- сервису;
- IP;
- интеграции;
- типу операции;
- стоимости ИИ-запроса;
- объёму файлов.

При превышении лимита возвращается `429 Rate Limit Exceeded` с допустимым временем повторной попытки.

Критические внутренние события не должны теряться из-за перегрузки внешней интеграции.



## 46. Наблюдаемость

Для каждого запроса и события используются:

- `request_id`;
- `correlation_id`;
- распределённая трассировка;
- метрики задержки;
- частота ошибок;
- глубина очереди;
- длительность фоновых заданий;
- стоимость ИИ-вызовов;
- доступность интеграций;
- доля ручных исправлений классификации.

Система должна предупреждать о:

- остановке синхронизации;
- росте ошибок;
- накоплении очереди;
- истечении токена;
- неработающем webhook;
- аномальном объёме скачиваний;
- серии неудачных входов;
- повторных финансовых событиях.



## 47. Версионирование и совместимость

Публичный контракт версионируется:

```text
/api/v1
```

Несовместимое изменение требует новой версии.

Совместимые изменения:

- добавление необязательного поля;
- добавление нового endpoint;
- добавление нового типа события с отдельной регистрацией.

Несовместимые изменения:

- удаление поля;
- изменение его смысла;
- изменение типа;
- изменение обязательности;
- переименование статуса;
- изменение финансового поведения.

Устаревшие версии имеют объявленный срок поддержки и план миграции.



## 48. Документация API

Для каждого endpoint должны быть описаны:

- назначение;
- права;
- параметры;
- тело запроса;
- ответ;
- ошибки;
- бизнес-правила;
- идемпотентность;
- события;
- пример;
- требования аудита;
- тестовый сценарий.

OpenAPI-спецификация должна генерироваться или проверяться в CI.



## 49. Тестирование

Обязательные уровни:

- unit-тесты бизнес-правил;
- contract-тесты API;
- integration-тесты базы и очередей;
- тесты webhook;
- тесты повторной доставки;
- тесты прав доступа;
- тесты финансовой идемпотентности;
- тесты офлайн-синхронизации;
- тесты миграций;
- security-тесты;
- end-to-end сценарии.

Критические end-to-end сценарии:

1. Сообщение прораба превращается в задачу и связывается с объектом.
2. Фотоотчёт прикрепляется к задаче и проверяется ответственным.
3. Заявка на материал проходит согласование, закупку, приёмку и выдачу.
4. Подотчётный расход получает чек, проверку и авансовый отчёт.
5. Просроченная задача вызывает запрос статуса и эскалацию.
6. Письмо заказчика регистрируется, классифицируется и попадает ответственному.
7. Дубликат webhook не создаёт повторную задачу или финансовую операцию.
8. Пользователь не может читать чужой проект.
9. ИИ не может отправить критическое сообщение без утверждения.
10. Отключение внешнего сервиса не приводит к потере входящих событий.



## 50. Минимальный API для MVP

Первая рабочая версия должна включать:

1. Авторизацию и роли.
2. Сотрудников.
3. Проекты и объекты.
4. Задачи и доказательства.
5. Мобильный ежедневный отчёт прораба.
6. Универсальный входящий поток.
7. Одну почтовую интеграцию.
8. Один официальный мессенджерный коннектор.
9. Файлы и медиавложения.
10. Предложенные действия ИИ.
11. Человеческое утверждение.
12. Уведомления.
13. Подотчётные расходы и чеки.
14. Заявки на материалы и базовые остатки.
15. Журнал аудита.
16. Утренняя и вечерняя сводки.

MVP не должен пытаться одновременно заменить все корпоративные системы. Сначала проверяется полный рабочий цикл на одном или двух реальных объектах.



## 51. Этапы подключения каналов

### Этап 1. Внутренняя CRM и мобильные формы

- объекты;
- ежедневные отчёты;
- фото и видео;
- подотчётные расходы;
- заявки на материалы.

### Этап 2. Корпоративная почта

- входящие письма;
- привязка к проектам;
- черновики ответов;
- регистрация документов.

### Этап 3. Один официальный мессенджерный канал

- классификация;

### Этап 4. Дополнительные CRM и приложения

### Этап 5. Расширенная автоматизация

## 52. Критерии готовности интеграции

Интеграция считается готовой только если:

- используется официальный и разрешённый механизм;
- определён владелец данных;
- права минимальны;
- секреты защищены;
- webhook проверяется;
- дубликаты обрабатываются;
- ошибки не теряются;
- есть журнал синхронизации;
- есть тест отключения внешнего сервиса;
- есть процедура повторного запуска;
- есть процедура отзыва доступа;
- данные правильно связываются с проектом;
- критические действия требуют утверждения;
- результат подтверждён на staging;
- документация обновлена.



## 53. Критерии приёмки API

API принимается как базовый контракт, если:

- все ключевые бизнес-сущности имеют понятные операции;
- права соответствуют ролям и объектам;
- критические действия защищены утверждением;
- финансовые и складские операции идемпотентны;
- входящие сообщения сохраняют источник и историю;
- файлы проходят безопасную обработку;
- ИИ-действия объяснимы и журналируются;
- интеграции построены через адаптеры;
- мобильная работа поддерживает нестабильную связь;
- ошибки имеют единый формат;
- события имеют версию и идентификатор;
- тестовые сценарии покрывают ключевые процессы;
- OpenAPI соответствует фактической реализации.



## 54. Главный сквозной сценарий

1. Прораб отправляет разрешённым способом сообщение, фото, видео или мобильную форму.
2. Коннектор проверяет источник и регистрирует событие.
3. Система связывает данные с сотрудником, проектом и объектом.
4. ИИ определяет: отчёт, задача, расход, заявка, риск или документ.
5. Создаётся предложенное действие с объяснением и источниками.
6. Если действие критическое, руководитель получает запрос на утверждение.
7. После утверждения система создаёт поручение или запись.
8. Ответственный сотрудник получает уведомление.
9. Контролёр исполнения запрашивает статус и доказательства.
10. Просрочка или препятствие эскалируются руководителю.
11. Результат попадает в карточку объекта, журнал и сводку.
12. Незавершённые вопросы переходят в план следующего дня.



## 55. Финальное правило

Badrudin AI OS должна объединять информацию и помогать выполнять работу, но не скрывать источник, не подменять ответственного человека и не совершать критические действия без предусмотренного утверждения.

Каждая интеграция должна отвечать на пять вопросов:

1. Откуда получены данные?
2. Кто имеет право их видеть?
3. Какое действие предлагается?
4. Кто должен его утвердить?
5. Как проверить фактический результат?

Если хотя бы на один вопрос нет надёжного ответа, автоматическое действие не выполняется.
