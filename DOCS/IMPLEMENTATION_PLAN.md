фундамента, на котором затем строится ядро CRM (Этап 2). План опирается только на
утверждённые решения `DOCS/DECISIONS.md` D-001…D-018 и утверждённые документы
`MASTER_SPECIFICATION.md`, `ARCHITECTURE.md`, `DATABASE.md`, `ROADMAP.md`,
`ACCESS_CONTROL.md`.

Документ не изменяет бизнес-требования. Границы работ ограничены Этапом 1; ядро
CRM (Этап 2) и бизнес-модули в план не входят.

## 3. Принципы реализации Этапа 1

- сначала контекст, затем действие; минимальный безопасный объём изменений
  (`CLAUDE.md` разделы 2, 9.2);
- API-first, модульность, аудит каждого действия, идемпотентность
  (`ARCHITECTURE.md` раздел 11);
- человек в контуре критических решений; действия уровня R4 не выполняются
  автоматически (D-001, D-002);
- Documentation First: документация обновляется вместе с кодом;
- безопасность по умолчанию: минимальные полномочия, запрет по умолчанию,
  секреты не в Git (`ACCESS_CONTROL.md` разделы 2, 24; D-008);
- персональные данные граждан РФ хранятся и обрабатываются в РФ (D-011).

## 4. Утверждённый стек и инфраструктура

- **Backend / API:** FastAPI (Python), Pydantic, SQLAlchemy, Alembic (D-004;
  `ARCHITECTURE.md` раздел 5.2).
- **Frontend:** Next.js (D-004; `ARCHITECTURE.md` раздел 5.1).
- **База данных:** PostgreSQL (D-004).
- **Оркестрация процессов:** n8n (D-004; `ARCHITECTURE.md` раздел 5.4).
- **Файловое хранилище:** MinIO (S3-совместимое) (D-008; `ARCHITECTURE.md`
  раздел 5.6).
- **Кэш и брокер очередей:** Redis (D-008).
- **Фоновые задачи:** Celery (worker + планировщик) на Redis (D-008).
- **Секреты:** переменные окружения сервера и `.env` вне Git; в репозитории —
  только `.env.example` (D-008; `ARCHITECTURE.md` раздел 12.3).
- **Размещение:** защищённый VPS/сервер в российском ЦОД, развёртывание через
  Docker Compose (D-007; `ARCHITECTURE.md` раздел 17).

## 5. Среды

| Среда | Назначение | Особенности |
|---|---|---|
| development | локальная разработка | тестовые данные, безопасные интеграции |
| test | автоматические тесты | тестовая БД, обезличенные данные |
| staging | приёмка перед запуском | структура как в production, обезличенные/тестовые данные |
| production | рабочая система | ограниченный доступ, бэкапы, мониторинг, защищённые секреты |

Staging обязательно отделён от production (`ROADMAP.md` раздел 7). Производственные
данные не копируются в разработку без обезличивания (`DATABASE.md` раздел 27).

## 6. Архитектура фундамента

Минимальный набор сервисов (Docker Compose, `ARCHITECTURE.md` раздел 17):

```text
frontend        (Next.js)
backend         (FastAPI)
worker          (Celery)
scheduler       (Celery beat)
postgresql
redis
n8n
object-storage  (MinIO)
```

Backend — единственная точка доступа к данным для интерфейса и агентов
(`ARCHITECTURE.md` раздел 5.2). Критическая бизнес-логика не хранится только внутри
n8n (`ARCHITECTURE.md` раздел 5.4).

## 9. Безопасность и доступ фундамента

- единая шкала риска действий R0–R4 (D-001; `ACCESS_CONTROL.md` раздел 9);
- действия уровня R4 выполняются только после подтверждения человека, с усиленной
  аутентификацией и полным аудитом (D-002);
- модель доступа RBAC + контекстные ограничения ABAC, изоляция по проектам и
  объектам (`ACCESS_CONTROL.md` раздел 4);
- MFA обязательна для критических ролей (`ACCESS_CONTROL.md` раздел 19);
- запрет самосогласования и разделение обязанностей (`ACCESS_CONTROL.md`
  разделы 10.1, 22);
- секреты — вне Git, в переменных окружения и защищённом хранилище n8n (D-008);
- журнал аудита `audit_events` неизменяем для обычных пользователей
  (`DATABASE.md` раздел 20);
- ПДн граждан РФ — в российском контуре; мягкое удаление и неизменяемый след
  (D-011).

## 11. Критерии готовности Этапа 1

Из `ROADMAP.md` раздел 7:

- система разворачивается повторяемым способом;
- миграции базы применяются автоматически;
- пользователь может войти и выйти;
- роли и области доступа работают;
- каждый запрос имеет трассировку;
- резервная копия создаётся и проверяется восстановлением;
- секреты отсутствуют в репозитории;
- staging отделён от production.

Дополнительно: журнал аудита фиксирует критические действия; сбой интеграции не
приводит к потере данных; критические действия требуют подтверждения человека
(D-002).

## 12. Действия, требующие подтверждения человека (R3/R4)

Следующие операции Этапа 1 не выполняются автоматически и требуют подтверждения
уполномоченного лица (владелец продукта / владелец инфраструктуры / владелец
безопасности — D-012):

| Действие | Уровень | Кто подтверждает |
|---|---|---|
| Развёртывание production-окружения (VPS в РФ, Docker Compose) | R4 | владелец продукта + владелец инфраструктуры |
| Назначение/изменение секретов production, изменение настроек безопасности | R4 | владелец безопасности |
| Восстановление БД из резервной копии (drill на test/staging — R3; на production — R4) | R3/R4 | владелец инфраструктуры (prod — + владелец продукта) |
| Выпуск домена и TLS-сертификата, публичный доступ к окружению | R3 | владелец инфраструктуры |
| Предоставление внешнего доступа (`project_access` внешним лицам) | R3 | владелец безопасности |
| Развёртывание/доступ к staging-среде | R3 | владелец инфраструктуры |

Массовое удаление, экспорт и изменение прав — R4 в любом окружении (принцип
`ACCESS_CONTROL.md` раздел 9.5, D-002); в Этапе 1 такие операции не
предусмотрены как автоматические.

## 13. Риски Этапа 1 и меры

Подмножество `ROADMAP.md` раздел 32, применимое к фундаменту:

- **Неофициальные интеграции** → только официальные API и каналы (D-014).
- **Утечка конфиденциальных данных** → минимальные права, шифрование, секрет вне
  Git, аудит (D-008, D-011).
- **Ошибки синхронизации/дубликаты** → идемпотентные ключи запросов
  (`ARCHITECTURE.md` раздел 13).
- **Слабая связь на объектах** → офлайн-черновики и безопасная синхронизация как
  требование к мобильным формам (реализация — Этап 3; в Этапе 1 закладываются
  идемпотентность и трассировка).
- **Размытая ответственность** → владельцы направлений назначены (D-012).

## 15. Что НЕ входит в Этап 1

- ядро CRM и бизнес-экраны (Этап 2);
- бизнес-модули (задачи как рабочий процесс, отчёты, склад, финансы) сверх схемы
  данных и каркаса;
- ИИ-агенты в продуктивной работе (только таблицы `ai_agents`, `agent_runs` и
  адаптер как каркас — D-010);
- живые интеграции почты/Telegram/бухгалтерии (только каркас коннекторов);
- автоматическая юридическая подпись (D-018).
