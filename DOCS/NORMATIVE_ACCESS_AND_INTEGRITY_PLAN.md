# NORMATIVE_ACCESS_AND_INTEGRITY_PLAN.md

**Проект:** Badrudin AI OS для ООО «Экстра-Элит»
**Статус:** проект на согласование (D-016, CLAUDE.md §8/§30). Реализация — поэтапно, отдельными PR, без слияния до подтверждения владельца.
**Назначение:** зафиксировать нормативную базу, матрицу должностных прав, разделение обязанностей, персональную аутентификацию, защиту документов от подделки, неизменяемый аудит и план миграций — как основу этапа 1 «фундамент доступа и целостности».

> **Дисциплина изменений (CLAUDE.md §2, §9.2):** документ описывает, что **переиспользуется** из уже реализованного и что **добавляется**. Проект не переписывается с нуля. Юридическая и нормативная применимость **не утверждается автоматически системой** — её подтверждают уполномоченные специалисты ООО «Экстра-Элит» (главный инженер, инженер ПТО, юрист).

### 1.3. Проверка актуальности (обязательное правило)

- Система **не считает** указанную редакцию последней. Для каждого документа заполняются `edition`, `amendment_no`, `effective_from/until`, `status`.

- При изменении нормы **старые записи и отчёты не переписываются** — они сохраняют ссылку на редакцию, действовавшую на дату выполнения работы (историческая привязка).

## 2. Нормативный профиль проекта (`project_normative_profiles`)

- Отдельная сущность на проект: набор применимых нормативов + применимая редакция + обязательность + пункты по видам работ.
- Не применять все нормативы автоматически ко всем объектам. Профиль **утверждает** главный инженер / инженер ПТО / юрист.
- Поля позиции профиля: `project_id · normative_document_id · applicable_edition · mandatory (bool) · work_types[] · special_requirements (проект/заказчик/экспертиза) · approved_by · approved_at`.
- Контрольные карты по видам работ (§ см. `PROCESS_CORE_PLAN.md`) ссылаются на позиции профиля как на нормативное основание.

## 3. Матрица должностей и прав (RBAC × ABAC × field-level × SoD)

### 3.1. Роли (расширение `dev_seed.json`)

`general_director · executive_director · chief_engineer · project_manager · site_manager (начальник участка) · foreman (прораб/производитель работ) · master (мастер) · pto_engineer · construction_control_engineer · hse_specialist (охрана труда) · supply (снабженец) · storekeeper (кладовщик) · mechanic (механик) · accountant (бухгалтер) · lawyer (юрист) · hr_specialist · clerk (делопроизводитель) · executor (обычный исполнитель) · external_contractor (ограниченный доступ)`.

### 3.2. Комбинированная модель (все проверки — на backend, §ниже)

- **RBAC** — право по должности (`permission` коды).
- **ABAC** — с учётом проекта/объекта/подразделения/стадии/суммы/уровня риска/назначения.
- **project scope** — только разрешённые проекты (`accessible_project_ids`).
- **record ownership** — доступ к своим/порученным записям.
- **field-level** — ограничение изменения отдельных полей (напр., прораб не меняет финансовые поля).
- **segregation of duties (SoD)** — разделение создания / исполнения / проверки / утверждения.

### 3.3. Ключевые должностные права (выдержка; полная матрица — в коде seed + сервисе прав)

| Роль | Может | Не может (SoD/field-level) |
|---|---|---|
| Генеральный директор | видеть всю организацию; назначать руководителей; утверждать критические (R4); принимать исключения | незаметно переписать утверждённый первичный документ |
| Исполнительный директор | операционное управление; назначения в пределах полномочий; согласование исключений | менять техзаключения профильных специалистов от их имени |
| Главный инженер | утверждать техрешения, нормативные профили, техкарты, техотклонения | проводить финансовые операции без отдельного права |
| Руководитель проекта | управлять своими проектами; ставить/распределять задачи; контроль сроков | утверждать собственный результат там, где нужна независимая проверка |
| Инженер ПТО | вести исполнительную документацию, реестры, акты, объёмы | подписывать за производителя работ / стройконтроль / руководителя |
| Прораб | производство работ; ежедневные отчёты; люди/техника/материалы/объёмы | менять финансовые документы и нормативный профиль |
| Инженер стройконтроля | проверки, замечания, решения по контролю | менять отчёт исполнителя от его имени; сам устранить и сам принять одно замечание |
| Специалист ОТ | проверки, предписания, инструктажи, замечания по безопасности | менять производственные объёмы и финансовые документы |
| Снабженец | заявки, запросы цен, закупочные документы | принимать материал на склад вместо кладовщика; утверждать свою закупку сверх лимита |
| Кладовщик | поступление, выдача, остатки | менять утверждённые заявки и договорные цены |
| Бухгалтер | счета, оплаты, бухгалтерские подтверждения | подтверждать физическое выполнение СМР |
| Юрист | договоры, письма, претензии, правовые заключения | менять производственные отчёты |
| Обычный исполнитель | видеть/принимать назначенные задачи; прикладывать результат | менять постановщика, срок, проверяющего, утверждённые документы |
| Внешний подрядчик | ограниченный доступ к назначенным процессам | всё прочее |

Все права **настраиваются администратором** (данные seed/БД), без изменения программного кода.

### 3.4. Разделение обязанностей (SoD) — правила-инварианты

- Создатель критического (R3–R4) результата **не может** сам его окончательно проверить/закрыть.
- Проверяющий замечание **не может** быть тем, кто его устранял.
- Инициатор закупки **не может** быть приёмщиком материала и утверждающим сверх лимита одновременно.
- Бухгалтер **не подтверждает** физическое выполнение работ; стройконтроль/ПТО **не проводят** платежи.

## 4. Персональная аутентификация и ключи доступа

### 4.1. Персональные учётные записи (расширение `employees`/`users`)

Запрещены общие логины/пароли, передача учётной записи, работа под чужим аккаунтом, подмена автора/исполнителя задним числом. Для пользователя хранятся: ФИО, должность, подразделение, руководитель, табельный номер, телефон, рабочая почта, доступные проекты, должностная роль, конкретные полномочия, дата начала/прекращения полномочий, статус учётной записи, назначенные средства аутентификации.

### 4.2. Средства аутентификации (поэтапно)

- **Есть сейчас:** личный логин + пароль (bcrypt), TOTP MFA (`mfa_secret`, обязателен для привилегированных ролей).
- **Добавляется (этап 1):** таблица `mfa_recovery_codes` (одноразовые, хэшированные) — резервные коды восстановления.
- **Добавляется (отдельный безопасный под-PR):** WebAuthn/FIDO2 passkey / аппаратный ключ — таблица `webauthn_credentials` (`credential_id`, `public_key`, `sign_count`, `aaguid`, `status active|suspended|revoked`, `registered_at`, `user_id`, `label`). Требует библиотеки WebAuthn и фронтенда `navigator.credentials`; закрытые ключи пользователя **на сервере в открытом виде не хранятся** (хранится только публичный ключ).
- **Единый постоянный текстовый «ключ» для всех сотрудников — запрещён.** Каждый ключ персональный, отзываемый, заменяемый, с датой регистрации и статусом; события — в журнал безопасности.

### 4.3. Отзыв доступа

При увольнении/отстранении/смене должности доступ **немедленно** отзывается: `user.status`, признак действия полномочий по дате, статусы ключей → `revoked`. Отозванный ключ входа не допускает вход (тест обязателен).

### 4.4. Разграничение «вход» ≠ «подпись»

Ключ входа в систему **не является** квалифицированной электронной подписью. Юридически значимое подписание — отдельный модуль интеграции ЭП (после определения вида подписи, сертификатов, оператора и внутреннего порядка ООО «Экстра-Элит»). Кнопка «Подписано» не заменяет подпись.

## 5. Защита документов и файлов от подделки

### 5.1. Обязательные атрибуты файла (частично уже в `File`)

автор загрузки; дата/точное время; организация; проект; объект; тип документа; версия; **SHA-256**; размер; MIME-тип; оригинальное имя; безопасное серверное имя (`storage_key`); статус проверки; история действий; связи с задачами/отчётами.

### 5.2. Инварианты целостности (добавляется этапом 1)

После утверждения документа: **запрет замены исходного файла**, **запрет физического удаления** (только архивирование), **только новая версия**; предыдущая версия сохраняется; исправление — отдельное действие с причиной, автором, датой и основанием.

Запрещено изменение задним числом: автора; фактической даты загрузки; исполнителя; согласующего; результата согласования; контрольной суммы; истории статусов.

Для критических документов — принцип двух лиц (один создаёт, другой проверяет, уполномоченный утверждает).

### 5.3. Проверки загрузки (частично есть)

тип и размер (`validate_upload`, `allowed_upload_mime`, `max_upload_bytes`), безопасное имя, отсутствие прямого доступа к чужим файлам (ABAC на выдачу), отметка отсутствия/изменения метаданных как фактор риска (не утверждать достоверность геометки/времени съёмки).

## 6. Неизменяемый аудит (расширение `AuditEvent`)

Для каждого действия фиксируется: пользователь; **его должность на момент действия**; дата/время; тип действия; объект действия; старое/новое значение; причина; **IP-адрес**; **идентификатор сессии**; результат проверки прав; **применённый уровень аутентификации** (пароль / MFA / passkey).

Журнал не редактируется через обычный интерфейс; руководитель может просматривать, но не переписывать. Финансовые/складские/юридические/управленческие записи не удаляются бесследно (архивирование, история статусов).

---

## 7. Перевод сотрудника / смена должности

Права зависят не только от роли, но и от **периода полномочий**:
- старые действия сохраняются за прежней должностью (историческая привязка в аудите);
- новые права действуют с указанной даты, старые прекращаются;
- незавершённые задачи официально **переназначаются**, переназначение — в аудит.

## 8. План миграций (этап 1 — фундамент доступа и целостности)

Предполагаемая цепочка (номера — после текущего head `0030`):
- `0031_identity_profile` — расширение `employees`/`users` (должность, подразделение, руководитель, табельный номер, период полномочий, статус) + `mfa_recovery_codes`.
- `0032_webauthn_credentials` — таблица passkey/FIDO2 (под-PR безопасной аутентификации).
- `0033_audit_hardening` — доп. поля `audit_events` (должность-на-момент, ip, session_id, auth_level, reason) + индекс.
- `0034_file_integrity` — поля/ограничения блокировки файла после утверждения (`locked_at`, `locked_by`, запрет удаления через сервис) + `is_archived` для документов.
- `0035_normative_registry` — `normative_documents` + `project_normative_profiles` (+ позиции).

Каждая миграция: описание, upgrade/downgrade, round-trip на SQLite, оценка риска. Денежные/юридические данные не теряются.

## 9. Список тестов безопасности (этап 1)

- сотрудник не видит чужие проекты; данные другой организации недоступны (есть база — расширить);
- исполнитель не редактирует чужую задачу; автор не принимает сам критический результат;
- прораб не редактирует финансовый документ; бухгалтер не принимает СМР; снабженец не принимает материал вместо кладовщика; ПТО не утверждает за стройконтроль (field-level + SoD);
- запрет замены утверждённого файла; запрет удаления первичного файла; новая версия не уничтожает старую;
- отозванный ключ/уволенный сотрудник теряет доступ;
- изменение роли не переписывает старый аудит; период полномочий учитывается;
- frontend-ограничение нельзя обойти прямым API-запросом (есть матрицы `test_access_control_matrix`/`test_write_access_matrix` — расширить на новые эндпоинты);
- права проверяются на уровне записи и изменяемого поля;
- нормативный профиль: норматив не применяется автоматически ко всем объектам; историческая редакция сохраняется.

## 10. Требует подтверждения специалистами ООО «Экстра-Элит»

Система — инструмент управления и контроля, **не замена** юридического заключения, строительного контроля, авторского надзора или решения ответственного специалиста. До перевода нормативов в `in_force` и включения обязательности требуется подтверждение:
1. **Актуальные редакции** всех перечисленных СП/СНиП/СанПиН/ГОСТ/приказов (главный инженер / ПТО / юрист).
2. **Применимость** каждого норматива к конкретным объектам и видам работ (нормативный профиль проекта).
3. **Пороговые суммы, сроки и уровни согласования** (R1–R4) — настраиваются, не зашиваются.
4. **Порядок электронной подписи** (вид, сертификаты, оператор) — до включения ЭП-контура.
5. **Кадровые данные и полномочия** (табельные номера, периоды полномочий, руководители).

## 11. Порядок реализации (PR-этапы; без слияния без подтверждения)

1. **PR-A (этот):** архитектурные документы — `NORMATIVE_ACCESS_AND_INTEGRITY_PLAN.md` + `PROCESS_CORE_PLAN.md`. Ожидает подтверждения владельца.
2. **PR-B — Этап 1 «Фундамент доступа и целостности»:** персональные аккаунты + профиль должности; recovery-коды MFA; должностные права (RBAC+ABAC+field-level+SoD, backend-проверки); проектные ограничения (есть) + владение записью; аудит-хардненинг; защита файлов (блокировка/версии/запрет удаления); нормативный справочник + профиль проекта; миграции 0031/0033/0034/0035; тесты безопасности; демо-логины должностей.
3. **PR-C — WebAuthn/passkey** (миграция 0032) — отдельный безопасный контур.
4. **PR-D — Единое процессное ядро** (workflow engine + Evidence Gate + исключения) — см. `PROCESS_CORE_PLAN.md`.
5. **PR-E — Ежедневный отчёт + ИИ-черновик (без автоутверждения) + «работы не велись».**
6. **PR-F — Контроль качества (контрольные карты по видам работ) + руководительские панели.**

На каждом этапе: миграции, backend/frontend тесты, ручная проверка сценария в браузере, зелёный CI, PR. **Слияние — только после подтверждения владельца.**
