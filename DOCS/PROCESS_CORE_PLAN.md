# PROCESS_CORE_PLAN.md — Единое процессное ядро ООО «Экстра-Элит»

**Статус:** проект на согласование (CLAUDE.md §8/§30). Реализация — отдельными PR, без слияния до подтверждения владельца.
**Назначение:** описать универсальный workflow engine, применяемый ко всем модулям; матрицу обязательных доказательств (Evidence Gate); уровни риска; запросы исключений; ежедневный отчёт объекта; контроль качества; поэтапный план и тесты.
**Связанный документ:** `NORMATIVE_ACCESS_AND_INTEGRITY_PLAN.md` (нормативы, права, аудит, ключи, целостность).

> **Принцип (требование владельца):** дисциплина и доказательность **без бюрократии ради бюрократии**. Дополнительные согласования — только там, где есть риск для безопасности, качества, денег, сроков, договорных обязательств или нормативного соответствия. ИИ **предлагает**, но не утверждает и не признаёт работу соответствующей норме — окончательное решение за уполномоченным специалистом.

---

## 0. Переиспользование (не переписывать проект)

Ядро **не заменяет** существующие модули, а даёт им общий процессный слой поверх уже реализованных сущностей:
- согласования — `Approval`/`ApprovalStep` (task.py);
- аудит — `AuditEvent` (audit.py);
- файлы/доказательства — `File` (SHA-256, метаданные) + `services.storage`;
- документы/версии — `Document`/`DocumentVersion`, `ExecutiveDocument`;
- доступ — `services.access` (проектный ABAC), RBAC;
- задачи как первый носитель процесса — `Task`/`TaskAssignment`/`TaskEvidence` (task.py) и модуль `task_control`.

Существующие модули (закупки, склад, техника, финансы, ПТО, стройконтроль/риски, инбокс, поручения, ежедневные отчёты) **подключаются к ядру постепенно**, начиная с поручений/задач как эталонного носителя.

## 1. Универсальный workflow engine

### 1.1. Модель `workflow_processes` (новая, обобщённая)

Единая запись процесса для любого вида работ. Поля:

`id · organization_id · project_id · site_id · process_kind · title · description · risk_level (R1..R4) · priority · status · overdue (вычисляемый признак) · blocked_reason · result · executor_comment · reviewer_comment · author_user_id (автор) · initiator_user_id (постановщик) · responsible_manager_id (ответственный руководитель) · primary_executor_id (основной исполнитель) · co_executor_ids[] (соисполнители) · watcher_ids[] (наблюдатели) · created_at · start_due_at (срок начала) · due_at (крайний срок) · accepted_at (дата принятия) · submitted_at (отправка на проверку) · completed_at · archived_at · reschedule_count · related_document_ids[] · related_material_ids[] · source_kind/source_id (связь с исходной сущностью модуля)`.

История изменений и согласований — через `AuditEvent` и `Approval`/`ApprovalStep` (не дублировать таблицы).

### 1.2. Состояния (единый жизненный цикл)

```
draft → pending_approval → approved → assigned → accepted → in_progress
     → submitted_for_review → (revision_required → in_progress) | completed → archived
```

Дополнительные состояния (не заменяют рабочий статус, кроме терминальных):
`cancelled · rejected · blocked · waiting_for_documents · waiting_for_external_party`.

**`overdue` — вычисляемый признак**, а не отдельный статус: хранится/считается одновременно с `in_progress`/`revision_required` (по `due_at < now` и статус не терминальный), чтобы не портить модель данных.

### 1.3. Правила переходов (инварианты, проверяются на backend)

- Назначение (`assigned`) делает **постановщик** (не сам исполнитель).
- Переход `assigned → accepted` — **только исполнитель**, отдельным действием «Принять в работу»; фиксируются дата, время, пользователь.
- Сотрудник видит только доступные проекты и назначенные ему процессы (ABAC + ownership).
- `submitted_for_review → completed` для R2–R4 — **независимая проверка** (SoD): исполнитель не закрывает сам (кроме явно разрешённых R1).
- Перенос `due_at` требует **причины**; для R3–R4 (или при `reschedule_count`/влиянии на связанные работы) — согласование руководителя. Все переносы, смены исполнителя и существенных условий — в аудит.
- Терминальные `completed|cancelled|rejected|archived` — далее только `archived` (архивирование вместо удаления).

### 1.4. Виды процессов (`process_kind`) — единый справочник

объекты/проекты · поручения/задачи · проектирование · стройпроизводство · стройконтроль · входной/операционный/приёмочный контроль · исполнительная документация · договоры/допсоглашения · закупки/заявки на материалы · склад/движение материалов · техника/механизмы · финансы/платежи · КС-2/КС-3/закрывающие · вх./исх. корреспонденция · претензионная работа · сотрудники/допуски · охрана труда · ежедневные отчёты · дефекты/замечания/предписания · согласования · риски · ИИ-предложения.

Каждый вид имеет: профиль риска по умолчанию (R1–R4), матрицу доказательств (§2), маршрут согласования, применимые контрольные карты (§5).

## 2. Evidence Gate — матрица обязательных доказательств

### 2.1. Модель

- `evidence_requirement_matrix` — настраиваемая матрица «вид процесса → обязательные доказательства» (`process_kind · evidence_type · required (bool) · condition · min_count · phase (before|during|after)`).
- `process_evidence` — фактически приложенные доказательства процесса (`process_id · evidence_type · file_id → File · note · captured_phase · added_by · added_at`). Файл — обязателен (карточка без файла не считается доказательством).

Типы доказательств: электронный оригинал · PDF · скан · фото · видео · акт · накладная · счёт · сертификат · паспорт качества · протокол испытаний · исполнительная схема · журнал работ · переписка · иной файл по матрице.

### 2.2. Правило гейта (инвариант)

`submitted_for_review`/`completed` **запрещены**, пока не приложены все обязательные доказательства по матрице процесса. Примеры матрицы:
- получение материала: накладная + фото материала (+ паспорт/сертификат при необходимости);
- выполненная работа: фото до/в процессе/после + объём + место;
- скрытая работа: фото до закрытия + необходимые акты;
- поломка техники: фото/видео + описание + показания + сведения о механизме;
- платёж: основание + договор/счёт/акт + согласование;
- исходящее письмо: проект письма + приложения + адресат + подтверждение отправки;
- приёмка результата подрядчика: документы + фото/видео + замечания;
- устранение замечания: фото до/после + комментарий проверяющего.

### 2.3. Запрос исключения (`evidence_exception_requests`)

Нельзя просто «поставить галочку, что документа нет». При отсутствии обязательного доказательства:
- исполнитель **обязан указать причину**;
- создаётся запрос на исключение (через `Approval`, тип `evidence_exception`);
- согласует **генеральный или исполнительный директор** (для технических вопросов — дополнительно главный инженер);
- решение, причина и согласующий — в **неизменяемом** журнале;
- система **явно помечает** результат как «принят без стандартного доказательства».

## 3. Уровни риска и «не зарываться в формальности»

| Уровень | Порядок |
|---|---|
| **R1** низкий | простое назначение + подтверждение исполнителем, без доп. согласования |
| **R2** средний | назначение → принятие → доказательство → проверка руководителем |
| **R3** высокий | предварительное согласование + обязательные доказательства + независимая проверка |
| **R4** критический | несколько согласующих + запрет самостоятельного закрытия + усиленный аудит (+ MFA для действия) |

К R3–R4 относятся процессы, связанные с: безопасностью; скрытыми работами; качеством несущих конструкций; крупными платежами; изменениями проекта; договорными обязательствами; приёмкой; претензиями; остановкой работ; существенным переносом сроков; действиями вовне от имени организации.

**Пороговые суммы, сроки и уровни согласования — настраиваемые** (таблица `risk_thresholds` по организации/проекту/виду процесса), не зашиты в код.

## 4. Сроки, просрочки, уведомления, эскалации

- Каждый процесс имеет `due_at`. Показатели: сколько осталось; приближение срока; просрочка; число переносов; кто и почему перенёс; влияние на связанные работы.
- Уведомления (внутренние, канал `in_app` — реальная внешняя отправка не включается без ключей и одобрения, §14): назначено · требуется принять · приближается срок · просрочено · возвращено · требуется согласование · документ отсутствует · ежедневный отчёт не предоставлен · отчёт без обязательного подтверждения · замечание не устранено · критический риск.
- Эскалации: непринятие в срок → постановщику/руководителю; критическая просрочка → руководителю; систематические нарушения → исполнительному и генеральному директору.
- Переиспуется существующий центр уведомлений (`/notifications`, `Notification`, `in_app`).

## 5. Строительный контроль и качество (контрольные карты)

- `quality_control_cards` — настраиваемые карты по видам работ: земляные работы; траншеи и крепление стенок; подготовка основания; песчаная подготовка; укладка трубопроводов; сварка/соединение полимерных труб; колодцы; обратная засыпка; послойное уплотнение; геодезическая разбивка; исполнительная съёмка; гидроиспытания; промывка/дезинфекция; ГНБ; входной контроль труб/материалов/оборудования; охрана труда; ПРР; пожарная безопасность; обращение с отходами; восстановление благоустройства; скрытые работы; приёмочный контроль.
- Поля карты: нормативное основание (ссылка на позицию нормативного профиля) · контролируемый параметр · допустимое значение · метод проверки · ответственное лицо · обязательный документ · обязательное фото/видео · необходимое измерение · дата/время проверки · применённый прибор · сведения о поверке прибора · результат · замечание · срок устранения · повторная проверка · итоговое решение уполномоченного специалиста.
- Замечания/предписания/дефекты → процессы вида `defect`/`prescription` с Evidence Gate «фото до/после + комментарий проверяющего» и SoD (устранял ≠ принимал).
- **ИИ только подсказывает** возможное отклонение; признать работу соответствующей норме — право уполномоченного специалиста.

## 6. Ежедневный отчёт объекта

- Переиспуется/расширяется существующий контур (`DailyReport`, `field_report`): фото/видео, выполненные работы, объёмы, участок, число работников, подрядчики, техника, поступившие/использованные материалы, погода, простои, препятствия, замечания, нарушения, происшествия, вопросы руководству, план на завтра.
- Медиа хранят: автор загрузки, дата/время, объект, привязку к отчёту, подпись/комментарий, SHA-256, оригинал, метаданные (если есть). **Достоверность геометки/времени съёмки не гарантируется** — отсутствие/изменение метаданных помечается как фактор риска.
- **ИИ:** анализ фото/видео, группировка по видам работ, черновик описания и отчёта, поиск расхождений и отсутствующих кадров, сопоставление с задачами/графиком/предыдущим днём, сводка директору с рисками/простоями/просрочками. **ИИ не утверждает отчёт** — подтверждает ответственный сотрудник (переиспуется honest-контур `agents`, D-010).
- Отсутствие фото/видео → отчёт нельзя отправить обычным способом; причина → запрос исключения → решение ген./исп. директора → аудит.
- День без работ → отчёт «Работы не велись» с причиной и подтверждением ответственного.

## 7. Отчётность с источниками (доказательность вывода)

Любой управленческий/ИИ-отчёт показывает источники: период · объект · использованные документы · использованные фото/видео · задачи/процессы-основания · отсутствующие данные · уровень уверенности · предупреждение о неподтверждённых сведениях. Руководитель открывает каждый вывод и видит первичный материал. **ИИ не выдаёт предположение за факт** (переиспуется KPI/аудит-контур).

## 8. Руководительские панели

- **Генеральный директор:** объекты; состояние проектов; исполнители; задачи в работе; непринятые; просрочки; ожидающие согласования; отчёты без фото/видео; запросы исключений; отсутствующие документы; риски; платежи; закупки; исполнительная документация; замечания стройконтроля; простои; ежедневная ИИ-сводка.
- **Исполнительный директор:** операционное состояние + приём предусмотренных исключений.
- **Руководитель проекта:** только свои проекты и подчинённые процессы.
- **Исполнитель:** назначенные задачи; сроки; обязательные подтверждения; комментарии; возвраты на доработку; история своих действий.

Панели строятся на уже имеющихся дашбордах (`/dashboard`, `/dashboard/digest`, `/kpi`, `/tasks/control`) + новые срезы процессного ядра.

## 10. Список тестов ядра

- назначение; принятие задачи (фиксация даты/пользователя); запрет принятия не-исполнителем;
- сроки и просрочка (вычисляемый признак `overdue`); перенос срока с причиной; согласование переноса для R3–R4;
- смена исполнителя (в аудит); возврат на доработку (`revision_required`);
- Evidence Gate: запрет `submitted_for_review`/`completed` без обязательных доказательств;
- запрос исключения: только ген./исп. директор согласует; результат помечается «без стандартного доказательства»; запись в аудит;
- SoD: автор не закрывает сам критический результат; устранявший замечание не принимает его;
- ежедневный отчёт: обязательность медиа; «работы не велись»; ИИ-черновик **без** автоутверждения;
- изоляция проектов/организаций; доступ к файлам; RBAC/ABAC/field-level на каждом эндпоинте (расширение матричных тестов).

## 11. Порядок реализации (PR-этапы; слияние — только с подтверждением владельца)

1. **PR-A (этот):** архитектурные документы.
2. **PR-B:** Этап 1 «Фундамент доступа и целостности» (см. связанный документ §11).
3. **PR-C:** WebAuthn/passkey.
4. **PR-D:** Ядро workflow (`workflow_processes`) + Evidence Gate + исключения + подключение к поручениям/задачам как эталон; API + минимальный экран; тесты.
5. **PR-E:** Ежедневный отчёт + ИИ-черновик + «работы не велись».
6. **PR-F:** Контроль качества (контрольные карты) + нормативная привязка + руководительские панели.

На каждом этапе: миграции; backend/frontend тесты; ручная проверка сценария в браузере; зелёный CI; PR. **Не сливать без подтверждения.**

## 12. Демонстрационный сценарий (для приёмки PR-D/E)

Генеральный директор ставит задачу руководителю проекта → руководитель назначает инженера → инженер **принимает** задачу → загружает документ и фотографии → ИИ формирует **черновик** отчёта → инженер подтверждает → руководитель проверяет → при необходимости **возвращает** → после исправления принимает → весь процесс виден директору. Демо-пользователи (логины будут выданы при реализации PR-B): генеральный директор, исполнительный директор, руководитель проекта, инженер ПТО, прораб, инженер стройконтроля, снабженец.
